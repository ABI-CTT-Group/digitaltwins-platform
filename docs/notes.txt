Good question — let me explain the "why" first, then we'll test it.

Why Keycloak for Airflow?

Two separate concerns:

Triggering — the portal backend calls Airflow's API to start workflows. That uses a service account (the AIRFLOW_USERNAME/AIRFLOW_PASSWORD env vars). Those runs will show up in Airflow attributed to that service account, regardless of which portal user initiated them.
Monitoring — when a human wants to come to /airflow and look at what's running, Keycloak means they use their platform identity rather than needing a separate Airflow account. Their Keycloak realm roles determine what they can see/do (Admin, Viewer, etc.).
So the two spaces are: portal service account (for programmatic triggering) and individual human users (for UI monitoring). They're intentionally separate.

So what if we wanted to start things on the api and not use that service account? Can we use the user's own account if both the platform and the airflow are integrated with the same keycloak?

Yes, absolutely — and it's actually the better design for user-initiated actions. Since both systems share the same Keycloak realm, the user's access token is already trusted by both.

How it would work:

User logs into the portal → browser holds a Keycloak JWT access token
Portal frontend sends that token to the portal backend with the request
Portal backend forwards it to Airflow's REST API as a Bearer token
Airflow validates it against Keycloak (same realm, same signing keys)
Airflow identifies the user from the token's claims — same preferred_username, same roles
DAG run is attributed to that actual user
What needs to change:

Airflow's REST API currently accepts session cookies and basic auth. To accept Keycloak Bearer tokens, you'd need to add a JWT validation auth backend — essentially a small Python module that:

Extracts the Authorization: Bearer <token> header
Validates the JWT signature against Keycloak's public keys (the jwks_uri)
Maps the claims to a FAB user (which our KeycloakSecurityManager already knows how to do)
When to use which:

Scenario	Use
User clicks "Run workflow" in portal	Forward user's token → attributed to them
Scheduled/automated triggers	Service account → clean separation
Background sync jobs	Service account
The nice thing is you already have all the pieces — KeycloakSecurityManager decodes the JWT and maps roles. The same logic could back the API auth. It's not a huge lift, but it's a deliberate decision about whether you want that attribution granularity.

One caveat: The exchange endpoint is a Flask blueprint. In Airflow 3, where exactly Flask blueprints get mounted depends on the internal wiring — the URL /keycloak/token/exchange might end up at a slightly different path. The test curl will tell us immediately. If it 404s, we check with docker compose exec airflow-apiserver airflow plugins to confirm it loaded, then look at the registered routes.
